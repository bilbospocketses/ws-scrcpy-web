# v0.1.30-beta.37 — Linux Service-Mode Full Smoke (Fedora)

**Goal:** The full Linux service-mode validation pass that was owed since beta.32 (Smoke 3-7 never ran end-to-end — we got pulled into the #27 apply bug at Smoke 2), now folding in the **beta.37** UX fixes and a forward section for the **Phase 2** service-mode update apply (item 39) once it ships.

**Scope:** Linux AppImage on **Fedora 44, SELinux enforcing** (the oracle for items 32/33 and the Phase-2 system-scope path). The Windows multi-user / MSI / tray coverage is a separate, still-valid checklist — see [`v0.1.25-beta.3.md`](./v0.1.25-beta.3.md); run that on a Win11 VM. This doc does NOT duplicate it.

**Prerequisites on the VM:**
- Hyper-V Fedora 44 VM, `getenforce` → **Enforcing**. Clean snapshot before installing.
- `libfuse2` present (`sudo dnf install fuse-libs` if the AppImage won't mount — pre-Velopack-1.1.1 gate, item 31).
- A terminal for `systemctl`, `journalctl`, `ausearch`, `sudo`.
- The published **v0.1.30-beta.37** AppImage: `WsScrcpyWeb-linux-beta.AppImage` (channel-suffixed — `package-linux.mjs` packs `--channel linux-beta`).

**How to read this:** each test has setup / action / expected. Check off as you go; if anything fails, note the test id and what diverged, capture `journalctl --user -u WsScrcpyWeb` (or system) + `ausearch -m avc -ts recent`.

**Manual-recovery if a stuck system service recurs** (from item 41):
```bash
sudo systemctl stop/disable/reset-failed WsScrcpyWeb.service
sudo rm -f /etc/systemd/system/WsScrcpyWeb.service
sudo rm -rf /opt/ws-scrcpy-web
sudo semanage fcontext -d '/opt/ws-scrcpy-web(/.*)?'
sudo systemctl daemon-reload
```

## Phase 2 — service-mode in-app UPDATE apply (item 39 — run ONLY once that ships)

> Not in beta.37. These are the oracle checks from `docs/plans/2026-06-02-linux-service-mode-phase2-update-apply.md`. Run on the beta that ships item 39.

- [ ] **User-scope update:** user-service installed → Settings → Updates → Apply (beta.N→N+1). The `--user` unit stops, the home `$APPIMAGE` swaps, the unit restarts on the **same** web port, the browser reconnects via the upgrading overlay. No prompt.
- [ ] **System-scope update (headless — the open risk):** system-service installed → Apply → **no polkit prompt** (root self-update) → `/opt` copy swaps, `restorecon` re-applies `bin_t`, unit restarts, **no AVC**. If SELinux blocks the `init_t`/transient-unit `/opt` write or relabel, a *narrow targeted* policy is needed (never broad `audit2allow`).
- [ ] **Out-of-cgroup survival:** the `systemd-run` apply helper survives `systemctl stop` of the unit it is restarting.
- [ ] **Settle timing:** the FUSE unmount settles within the helper's 15s swap-retry window (widen if the VM is slow).
